# DukaanAI

> **Voice-first AI for kirana stores.** Designed for owners who already talk to suppliers, customers, and staff all day — and won't learn yet another app.

---

## The core reframe

Every existing tool (Khatabook, Vyapar, BharatPe) treats *"owners are not tech-savvy"* as a bug to onboard around. DukaanAI inverts the problem: **be a colleague they talk to, not an app they learn.** Owners speak in Tamil / Hindi / English / code-mix; DukaanAI parses, acts, and confirms.

That's the design principle the entire architecture flows from.

## Running the demo

### Prereqs
- Python 3.10+
- (Optional) An Anthropic API key — without it, the app runs in **demo mode** with heuristic fallbacks. Reviewers can run the whole flow offline.

### Install
```bash
cd dukaan_ai
pip install -r requirements.txt
python seed_data.py          # generates data/ JSON files
```

### Run
```bash
# Optional: set API key for live LLM mode
export ANTHROPIC_API_KEY=sk-ant-...

# Optional: override model (default: claude-sonnet-4-5)
# export DUKAAN_MODEL=claude-opus-4-5

streamlit run app.py
```

The UI opens at `http://localhost:8501`. Pick a scenario from the sidebar or speak/type your own.

## 60-second walkthrough for reviewers

The demo is structured around three scenarios mapped to three of the five core agents:

1. **Voice → Sale logged.** Type *"Anna do kilo sugar customer ko de diya UPI mein"* into the chat. Watch the Voice Parser produce structured JSON (intent + items + payment mode), then auto-log because trust Level 2 is set.

2. **Forecast.** Click the *"Show me what to reorder"* shortcut. The Replenishment Forecast Agent reads 30 days of sales, applies the festival multiplier for Eid al-Adha (5 days out at the time of this build), and returns a priority-ranked reorder list with reasoning per SKU.

3. **PO draft.** From the forecast, click *"Draft PO to Anna Wholesale"*. The Supplier PO Agent produces an owner-facing summary, a supplier-facing WhatsApp message, and a flag if the value crosses the trust-level ceiling.

A fourth scenario (Invoice generation) is triggered by saying *"Rajesh Traders ko 5 kg sugar aur 2 litre tel credit pe"* — the Invoice Agent generates a GST-compliant invoice including CGST/SGST split logic.

## Architecture in one paragraph

A **Conversational Router** (LLM) sits in front of four **specialist agents** (Voice Parser, Replenishment Forecast, Invoice, Supplier PO). Each agent has its own versioned system prompt and a Pydantic schema for input/output — so every LLM response is validated before it touches state. A **Trust Ladder** governs autonomy: new stores require confirmation for every action; trusted stores get silent auto-execution with a 30-second undo. Demotion is automatic if rejection rates spike. See `docs/architecture.md` for the diagram.

## Why this design over alternatives

| Alternative considered | Why we rejected |
|---|---|
| Single monolithic LLM prompt | One prompt change risks every flow. No per-agent evals. |
| Pure rules engine (no LLM) | Cannot handle code-mix Tamil/Hindi voice — the entire UX premise |
| Wrapper on Khatabook/Vyapar | Locks us into their data model + onboarding model (the thing we're inverting) |
| WhatsApp-only (no PWA) | WhatsApp Business API doesn't allow rich inline confirmations |

## What's hardcoded for the demo vs designed for production

| Concern | Demo | Production plan |
|---|---|---|
| Voice → text | Typed input (Streamlit doesn't ship STT) | On-device Whisper via the mobile PWA; fallback to Sarvam AI for code-mix |
| Auth | None | Phone-OTP login; store_id from JWT |
| Storage | JSON files in `data/` | Postgres + Snowflake for analytics |
| LLM calls | Direct Anthropic SDK, claude-sonnet-4-5 | Same, with prompt-version pinning + per-agent eval harness |
| Trust ladder | Hardcoded `trust_level=2` | Live updates in `D6: Trust + Audit Log` |
| Festival calendar | Static JSON | DB-backed, owner can add regional festivals |
